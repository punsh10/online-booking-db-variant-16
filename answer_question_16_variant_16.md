Контрольный вопрос №16

Олейников М.В. 454гр.

Для каких столбцов вашей схемы вы создали индексы, кроме первичных и внешних ключей? Обоснуйте выбор каждого индекса.

В схеме базы данных для коворкинг-центра я создал три дополнительных индекса помимо первичных ключей (PRIMARY KEY) и индексов, которые автоматически создаются для внешних ключей (FOREIGN KEY).
Индекс 1: idx_bookings_start_time на столбце start_time таблицы bookings.
Этот индекс нужен для быстрого поиска свободных временных интервалов при создании нового бронирования. Когда клиент хочет забронировать место, система должна проверить, не занято ли оно в выбранное время. Запрос будет содержать условие WHERE start_time >= '...' AND end_time <= '...'. Без индекса MySQL пришлось бы просматривать всю таблицу bookings, что при большом количестве записей работает медленно. Индекс по start_time позволяет мгновенно найти нужные строки. Кроме того, этот индекс ускоряет ключевой запрос моего варианта — популярность типов мест по дням недели, потому что в нём используется функция DAYOFWEEK(start_time) при группировке.
Индекс 2: idx_workspace_type на столбце workspace_type таблицы workspaces.
В интерфейсе коворкинга обязательно есть фильтр по типу места: пользователь может выбрать "показать только переговорные" или "показать только офисы". Запрос будет выглядеть так: SELECT * FROM workspaces WHERE workspace_type = 'переговорная'. Без индекса MySQL будет перебирать все строки таблицы workspaces. С индексом поиск выполняется мгновенно. Также этот индекс помогает в запросе про популярность типов мест по дням недели, так как группировка идёт по полю workspace_type.
Индекс 3: idx_discounts_active на столбце is_active таблицы discounts.
Когда клиент делает бронирование, система должна проверить, есть ли у него активная скидка. Запрос выглядит так: SELECT * FROM discounts WHERE client_id = ? AND is_active = TRUE. Индекс по is_active помогает быстро отсеять неактивные скидки. Это особенно полезно, когда в таблице много старых записей с истекшими скидками.
